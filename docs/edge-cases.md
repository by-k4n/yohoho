# yohoho — edge-case & resilience matrix

> Generated from an adversarial 7-surface sweep (149 cases: 16 critical, 60 high, 60 medium, 13 low).
> Companion to `DESIGN.md`. Each case maps to one of the nine resilience primitives (§ Resilience in DESIGN.md).

## The nine load-bearing primitives

- **P1 — generation-id + discard token**: a per-session monotonic id stamped at record-start, checked before *every* side effect (paste, clipboard, history, UI terminal event).
- **P2 — empty/silence/zero-length guard**: RMS floor pre-recognize + `result.strip()` post-recognize gate all side effects.
- **P3 — clipboard critical section**: `save→set→paste→(restore)` as one controller-owned `try/finally`, entered only with non-empty text after the abort check.
- **P4 — no clipboard auto-restore by default**: leave the transcript on the clipboard (config toggle to restore-with-delay).
- **P5 — paste is best-effort, transcript is never lost** + focus-token captured at record-stop (degrade to clipboard+history on any paste failure).
- **P6 — distinct error/aborted UI terminal states**, marshaled through a single Tk `queue+after()` pump; no worker thread touches Tk.
- **P7 — listener + permission supervisor**: heartbeat, re-arm dead listener, live macOS TCC probe + interpreter-identity check.
- **P8 — per-recording audio stream** bound to an explicit device index, opened/closed per session (no resident stream).
- **P9 — structural transcript-leak prevention**: `TranscriptText` wrapper + handler filter + a test asserting no log line contains transcript text.


## 1. Recording lifecycle & cancel/abort  (23 cases · 2 critical · 9 high)

- **[CRIT] `no-cancel-gesture-defined`** (controller) — There is no defined gesture to abort a dictation the user regrets, so the only way out is to finish recording and let unwanted text paste into the focused app.
  - **Handling:** Define an explicit CANCEL gesture available in BOTH recording and transcribing states. Primary: Esc key (registered via the same pynput listener while a session is active, torn down on idle). Secondary: a second press of the hotkey CANCELS only if a modifier is held (e.g. hotkey+Shift) OR a click anywhere on the always-on-top panel. Cancel must transition controller to a CANCELLING state that: stops the recorder, discards the buffer, and for transcribing sets a discard flag so the recognize() result is dropped. Never synthesize a paste on the cancel path.
- **[CRIT] `cancel-during-blocking-recognize`** (controller) — recognize(float32_16k) is a single blocking C/ONNX call with no cancellation token, so a cancel issued mid-transcription cannot interrupt the call; the result must be discarded after it returns.
  - **Handling:** Do NOT try to kill the engine thread. Make cancel cooperative: set an atomic `discard_token`/generation-id captured when the session started. When recognize() returns, the engine thread checks `if token != current_generation: drop result, do not enqueue paste`. The UI immediately flips to idle on cancel so the user gets instant feedback even though the engine is still busy. Document that recognize() for short dictation is ~1–2s so the worst-case 'busy' window is small; for long clips show that cancel is honored on return.
- **[HIGH] `double-toggle-reentrancy`** (controller) — Rapid double-press of the toggle hotkey (or key-repeat / bounce) can fire start→stop→start within milliseconds, racing the recorder open/close and the state machine.
  - **Handling:** Serialize all hotkey events through a single thread-safe controller queue and gate transitions with a lock + the explicit state machine; ignore toggles that don't match the current legal transition. Add a small debounce (~250–300ms) on the toggle in the chord matcher to swallow key-repeat/bounce. Make recorder open/close idempotent and await full stream teardown before honoring the next start.
- **[HIGH] `new-recording-while-transcribing`** (controller) — User presses the hotkey to start a NEW dictation while the previous clip is still in the transcribing state (engine thread busy with recognize()).
  - **Handling:** Define an explicit policy: while in transcribing/inserting, a start request is QUEUED (recording can begin immediately and capture audio, but its transcription waits its turn — single engine worker, FIFO) OR rejected with a brief 'busy' flash on the panel. Recommended: allow the new recording to start (mic capture is cheap) but funnel all recognize() calls through one serialized worker so pastes stay ordered and CPU isn't thrashed. Tag each job with a generation id so a later cancel can target the right one.
- **[HIGH] `quit-daemon-while-recording`** (daemon) — Daemon receives `yohoho stop` / SIGTERM / tray-quit while actively recording, with an open PortAudio stream and the pynput + Tk loops running.
  - **Handling:** Implement clean-shutdown semantics: install a signal handler that transitions to CANCELLING, then a deterministic teardown order — (1) stop/abort the sounddevice stream and close it, (2) stop the pynput listener, (3) set discard flag so any in-flight recognize() result is dropped, (4) quit the Tk mainloop via its queue, (5) release single-instance lock/PID file, (6) restore clipboard if we had overwritten it. Bound teardown with a timeout and os._exit() as last resort so stop never hangs.
- **[HIGH] `quit-daemon-while-transcribing`** (daemon) — Daemon is told to quit while the engine thread is inside the blocking recognize() call.
  - **Handling:** On quit set the discard flag and detach the engine thread as a daemon thread (Python daemon=True) so the process can exit without joining it; the result is dropped because discard is set, and no paste is enqueued once shutdown begins. Guard the paste/clipboard-restore step with an `if shutting_down: abort` check. Ensure the engine thread holds no un-restored clipboard state at the moment it might be abandoned.
- **[HIGH] `no-input-device`** (recorder) — Recording is triggered when there is no usable input device (headless machine, all mics disabled, or sounddevice default device index is invalid).
  - **Handling:** Wrap stream open in try/except for PortAudioError/sounddevice errors; on failure stay in idle, flash a clear panel message ('no microphone'), log the device enumeration + error, and keep the listener alive. Validate device availability at `setup` and at daemon `start`, and re-query devices lazily at each record start rather than caching a stale device index.
- **[HIGH] `mic-unplugged-mid-recording`** (recorder) — The active capture device disappears mid-recording (USB mic unplugged, Bluetooth headset disconnects, dock removed).
  - **Handling:** Detect device-removed errors in the audio callback / via a status flag; on detection, auto-stop the recording, transition out of REC, surface a 'mic disconnected' panel message, and either transcribe whatever was captured before the drop (if non-trivial) or discard if empty. Never leave the stream half-open — always close it in a finally. Log the event.
- **[HIGH] `cancel-leaves-clipboard-dirty`** (injector) — Cancel (or quit) happens during the inserting phase between 'set clipboard to text' and 'restore prior clipboard', or the paste chord fires but restore is skipped on the cancel path.
  - **Handling:** Treat the clipboard save/restore as a guarded critical section with try/finally: always restore the saved clipboard in finally, even on cancel/exception/shutdown. If cancel arrives before the paste chord is synthesized, restore immediately and never paste. Persist the saved-clipboard reference on the controller (not a local) so the shutdown handler can also restore it. The ~80–120ms sleep window before restore must be inside the same finally-protected block.
- **[HIGH] `panel-stuck-after-failed-session`** (ui) — An exception anywhere in the recording→transcribing→inserting pipeline leaves the Tk always-on-top panel showing REC/transcribing forever because the 'idle' UI event was never marshaled.
  - **Handling:** Wrap each worker thread's body in try/except/finally that ALWAYS enqueues an 'idle' UI event and a recorder-stop on exit, regardless of outcome, via the Tk queue+after() marshal. The controller's transition-to-idle (panel hide, mic release) must be in a finally block. Add a watchdog: if the panel has been non-idle longer than max-record-duration + a recognize() timeout, force it back to idle and log.
- **[HIGH] `hotkey-listener-thread-dies`** (daemon) — The pynput keyboard.Listener thread dies (uncaught exception in the on-press callback, or macOS revokes Input Monitoring mid-session) leaving the toggle unresponsive.
  - **Handling:** Wrap the on-press/on-release callbacks so no exception escapes (catch, log, continue). Run a supervisor that detects a dead listener thread and restarts it; if restart fails (e.g. permission revoked), surface it via tray/panel and `yohoho status`. On macOS, periodically/again verify kTCCServiceListenEvent is still granted since revocation fails silently, and reflect loss of permission in status.
- **[MED] `default-device-switched-mid-recording`** (recorder) — OS switches the default input device mid-recording (plugging in AirPods/headset, or another app grabs exclusive mic access) without erroring the stream.
  - **Handling:** Bind the stream to an explicit device index at open (don't follow 'default' live). If the captured RMS stays at near-zero for the whole clip despite the user speaking, treat as a likely device issue: surface a 'no audio captured — check microphone' hint instead of silently pasting empty text. Re-resolve the default device only at the start of each new recording.
- **[MED] `silence-near-zero-clip`** (recorder) — Clip contains only silence or near-zero amplitude (user pressed the hotkey but didn't speak, mic muted at hardware/OS level, or wrong device).
  - **Handling:** Before calling recognize(), check captured peak/RMS against a floor; if below threshold, skip transcription entirely, show a brief 'no speech detected' on the panel, do NOT paste, and do NOT write a history entry. After recognize(), also guard against empty/whitespace-only results: if result.strip() is empty, suppress the paste. Make the floor configurable.
- **[MED] `zero-length-clip`** (controller) — A 0-length (or sub-frame) audio buffer reaches the engine — e.g. start and stop fired within one audio block, or the stream never delivered a callback before stop.
  - **Handling:** Guard the engine entry: if the concatenated buffer has fewer than a minimum sample count (e.g. < ~100ms of audio), short-circuit to idle without calling recognize(), no paste, optional 'too short' flash. Protect RMS/normalization against zero-length and all-zero arrays. Treat 0-length as a clean no-op, not an error.
- **[MED] `extremely-long-recording-memory`** (recorder) — User starts recording and forgets to stop (toggle mode), or dictates a very long monologue, so the in-memory int16 buffer grows unbounded.
  - **Handling:** Enforce a configurable max recording duration (e.g. default 5 min) that auto-stops with a panel notice. Track elapsed time on the recorder and surface it in the existing MM:SS timer; warn (color change) as the cap approaches. Optionally cap total buffered samples and stop early. For long clips ensure the progress bar's asymptotic ease still animates so it never appears frozen.
- **[MED] `vad-autostop-premature`** (recorder) — In voice_activity_detection mode, VAD auto-stops on a natural pause (the user is mid-thought) and triggers transcription before they finish.
  - **Handling:** Make the VAD trailing-silence timeout configurable with a sane default (e.g. ~800ms–1.5s) and document the tradeoff. Use webrtcvad with hangover frames. Provide a clear mode distinction in setup: toggle mode never auto-stops; VAD mode auto-stops on silence. Show a subtle indicator when VAD is the active stop reason so the behavior is predictable.
- **[MED] `vad-never-stops`** (recorder) — In VAD mode, continuous background noise (fan, music, café) keeps VAD in 'speech' state so it never auto-stops.
  - **Handling:** Apply the same hard max-duration cap to VAD mode, and always keep the toggle-hotkey + Esc cancel active during VAD recording so the user can manually stop/cancel a runaway VAD session. Log VAD stop-reason (silence vs cap vs manual) for observability.
- **[MED] `system-sleep-during-recording`** (recorder) — The machine sleeps (lid close / idle / power loss to USB) while recording, then wakes later.
  - **Handling:** On the recorder, detect an abnormally large wall-clock gap between audio callbacks or a stream-error after resume and treat it as invalidation: stop/close the stream, drop the (now discontinuous) clip, reset to idle, and show 'recording interrupted'. Subscribe to OS sleep/wake notifications where available to proactively cancel any active recording on sleep. Always re-open a fresh stream for the next recording rather than reusing a pre-sleep handle.
- **[MED] `system-sleep-during-transcribing`** (controller) — Machine sleeps while the engine thread is inside recognize(), or right before the paste fires.
  - **Handling:** On wake after sleep, if a transcription completed during/after the sleep boundary, do NOT auto-paste blindly — set the discard flag on sleep so any pending result is dropped, and optionally surface it via history/notification instead of pasting into an unknown focus target. Re-check that the daemon is not at a lock screen before synthesizing a paste.
- **[MED] `cancel-during-recording-start-race`** (controller) — Cancel/stop is pressed in the brief window after start was accepted but before the PortAudio stream actually finished opening (open is not instantaneous).
  - **Handling:** Model an explicit STARTING substate; a cancel received in STARTING sets a flag so that when the stream open completes, the recorder immediately closes it and goes to idle (open-then-close), rather than dropping into RECORDING. All open/close paths must be reentrancy-safe and final state must be consistent with the last user intent.
- **[MED] `esc-swallowed-from-foreground-app`** (config) — Esc as the cancel gesture conflicts with the focused app, which also wants Esc (vim normal mode, modal dialogs, IDE), or pynput can't reliably suppress it.
  - **Handling:** Only listen for Esc while a yohoho session is active (recording/transcribing); register and unregister that listener around the session so it never interferes at idle. Document that Esc is the cancel key only during dictation. Provide an alternative cancel that doesn't collide (click the panel, or a configurable cancel hotkey) and let the user pick during setup. Don't rely on globally suppressing Esc.
- **[MED] `no-history-entry-on-abort-paths`** (history) — Cancelled, discarded, silence-only, and failed transcriptions must be consistently excluded from (or distinctly marked in) the history feature, and a successful-but-cancelled-before-paste result is ambiguous.
  - **Handling:** Define a clear history-write policy keyed off final controller outcome: write an entry only on successful paste; for cancelled-after-recognize results, optionally store under a 'discarded/recovered' bucket (not the main timeline) so the user can recover text they cancelled by accident, but never store empty/silence/error results. Tag each entry with outcome (pasted | discarded | failed) so the UI can filter.
- **[MED] `insufficient-cancel-abort-logging`** (logging) — Cancel/abort/error paths are exactly the events that are hardest to reproduce, so without structured logging the user can't report why a session vanished, got stuck, or pasted nothing.
  - **Handling:** Emit structured, leveled logs (to the per-user data dir, rotating) for every lifecycle transition and abort reason: session id/generation id, state transitions, stop-reason (manual | vad-silence | max-duration | device-error | cancel | shutdown | sleep), captured sample count + peak RMS, recognize() duration, paste/clipboard-restore success, and any caught exception with traceback. Surface a one-line summary in `yohoho status`. NEVER log transcript text or raw audio by default (privacy is the product) — gate that behind an explicit debug flag.


## 2. Transcription engine  (18 cases · 1 critical · 8 high)

- **[CRIT] `model-never-downloaded-on-start`** (engine) — Daemon `start` is run but the 630-660MB model was never fetched (setup skipped, or only the npm/uv bootstrap ran and the lazy first-setup download never happened).
  - **Handling:** Never let the engine trigger a network download. The engine `load()` must call `onnx_asr.load_model(...)` with HF local-files-only semantics (set HF_HUB_OFFLINE=1 around load, or pre-resolve the snapshot path) and raise a typed `ModelNotInstalled` error. The controller treats this as a hard precondition: at daemon `start` verify the model is present before entering `idle`; if missing, refuse to arm the hotkey and surface a distinct panel/tray state (`model not installed — run yohoho setup`). All large downloads happen only in `setup` with a real progress UI, never on the hotkey path.
- **[HIGH] `partial-interrupted-download`** (setup) — Model download was interrupted (Ctrl+C during setup, daemon killed, laptop slept, network dropped) leaving incomplete files or stale HF `*.incomplete` blobs in the cache.
  - **Handling:** After any download, run an integrity check before declaring the model ready: verify expected files exist (encoder/decoder/joiner int8 .onnx, vocab/tokenizer, config) and that HF blob sizes match snapshot metadata; treat any `.incomplete`/partial blob as not-installed. Persist a `model_ready` marker (with model id + revision) in the data dir written only after a verified load succeeds once. `setup` should use resumable download (huggingface_hub supports it) and offer `--repair` that clears the snapshot and refetches.
- **[HIGH] `corrupt-cache-load-failure`** (engine) — Model files present but corrupt (bad sectors, antivirus quarantine/truncation, OneDrive/iCloud cloud-placeholder dehydration leaving a 0-byte stub, or a botched int8 variant).
  - **Handling:** Wrap load in a typed `ModelLoadError` and validate eagerly at daemon start (load the session once, not lazily on first hotkey). On load failure: log full traceback + onnxruntime version + file sizes, set a degraded `engine_unavailable` state, surface a tray/panel message with a one-click `yohoho setup --repair`, and keep the hotkey listener alive but no-op (so the user isn't stuck with a dead daemon). Default the data dir to a non-synced local path to avoid cloud-placeholder corruption.
- **[HIGH] `onnxruntime-provider-init-error`** (engine) — onnxruntime native init fails: missing/incompatible VC++ redist on Windows, glibc/AVX CPU-feature mismatch, a requested GPU/CoreML provider that isn't available, or the documented import-order issue (onnxruntime must import before Qt-style native libs).
  - **Handling:** Force CPUExecutionProvider explicitly (v1 is CPU-only) and never request GPU/CoreML on the hotkey path; if a provider is unavailable, fall back to CPU with a logged warning rather than crashing. Keep `import onnxruntime` first in the engine module. In `setup`/preflight, detect missing VC++ redist (Windows) and AVX support and emit an actionable error. Catch provider init in a typed `EngineInitError` routed to the degraded state, not a daemon crash.
- **[HIGH] `oom-during-recognize-lowram`** (engine) — recognize() peaks RAM beyond free memory on a low-RAM (4-8GB) machine — resident model (~700MB-1GB) plus per-call activation buffers that scale with audio length — and the OS OOM-kills the daemon or thrashes swap.
  - **Handling:** Bound the input: enforce a max recording length (e.g. 60-120s) and chunk longer audio internally instead of one giant tensor. Cap onnxruntime intra-op threads (1-2) to limit peak memory and CPU contention. On restart, detect the OOM-kill (daemon was alive, now gone, last log line was 'transcribing') and log it as a likely OOM with the audio duration. Since OOM-kill can't be caught in-process, make clipboard restore crash-safe: persist the saved clipboard to a temp file before paste so a watchdog/next-start can restore it. Document a minimum-RAM requirement and check available RAM in `setup`.
- **[HIGH] `recognize-hang-or-pathologically-slow`** (controller) — A single recognize() call hangs or runs pathologically slowly — a degenerate/NaN audio buffer, an onnxruntime deadlock, CPU pinned by another process, or fp32-fallback pushing a 60s clip past tens of seconds.
  - **Handling:** Run recognize() on the dedicated worker with a watchdog timeout proportional to audio length (e.g. max(8s, 6x duration), capped, absolute ceiling ~90s). On timeout: abandon the result, return controller to `idle`, show a transient `transcription timed out` message, log duration + clip length + CPU load. Because Python can't safely kill a thread blocked in native ORT, isolate recognize in a separate engine subprocess the daemon can hard-kill and respawn (also caps blast radius of an ORT crash/OOM). At minimum, tag results stale so a late-returning call is discarded.
- **[HIGH] `concurrent-recognize-calls`** (controller) — Two recognize() calls overlap: user toggles stop while recognize is mid-flight, then presses again and dictates a second clip before the first returns (or a sticky/repeating hotkey fires two stops).
  - **Handling:** Serialize the engine: exactly one inference at a time via a single-worker queue (or a lock around recognize()). Enforce the controller state machine so a hotkey press during `transcribing` is ignored or queued, never run concurrently. Tag each job with a monotonic id; drop results whose recording was cancelled or superseded. Make clipboard save→set→paste→restore a single critical section so two insertions can't interleave.
- **[HIGH] `empty-transcript-result`** (injector) — recognize() returns an empty string or pure whitespace — user pressed the hotkey but didn't speak, mic was muted/zero-gain, captured only silence/noise, or VAD trimmed everything.
  - **Handling:** After recognize(), if the trimmed result is empty, short-circuit BEFORE touching the clipboard: do not save/replace/restore, do not synthesize paste. Transition to `idle`, complete the progress bar to 100% (so the UI still finishes cleanly), and optionally flash a subtle `no speech` hint. Only ever enter the clipboard critical section when there is non-empty text to insert.
- **[HIGH] `cancel-midstream-during-transcribe`** (controller) — User hits cancel/Esc (or closes the panel) while recognize() is already running on the worker thread — the explicitly-requested mid-stream cancel.
  - **Handling:** Make cancel set a `cancelled` flag on the current job id; when recognize() returns, the controller checks the flag and DISCARDS the result — no clipboard touch, no paste — going straight to `idle` and hiding the panel. Since native ORT can't be force-interrupted, for true immediacy run the engine in a separate subprocess the daemon can hard-kill on cancel and respawn (also solves hang/OOM isolation). Either way, guarantee a cancelled job never reaches the injector.
- **[MED] `missing-int8-quant-files`** (engine) — The int8 quantized ONNX variant is absent (repo revision only has fp32, or the int8 filenames onnx-asr expects for quantization='int8' aren't in the snapshot).
  - **Handling:** Pin the model repo revision (commit SHA), not just the name, so the int8 files are deterministic. After load, assert the active quantization is actually int8 (check loaded file names) and log a warning if a fallback occurred. If int8 is unavailable for the pinned revision, fail `setup` explicitly rather than silently shipping a 2x-heavier model.
- **[MED] `disk-full-mid-download`** (setup) — Disk fills during the 630MB download or during HF's copy-from-blob-to-snapshot step (which transiently needs ~2x space).
  - **Handling:** Preflight `setup` with a free-space check (require headroom, e.g. ~1.5GB to cover blob + snapshot copy) and refuse with a clear message if short. Catch ENOSPC during download, clean up partial files, and tell the user how much space is needed and where the cache lives. Make the cache/data dir location configurable so users can target a larger volume.
- **[MED] `cold-start-first-call-latency`** (engine) — First recognize() after daemon start pays cold-start cost — lazy session warmup, OS page-faulting the 630MB model from disk, allocator warmup — making the first dictation visibly slower than steady-state.
  - **Handling:** Eagerly load the model and run a warmup recognize() on a short silent buffer at daemon start (on the worker thread, off the UI) so the first real call is hot. Gate the hotkey/`idle` state on warmup completion, or allow recording but show a brief `loading model…` if a hotkey arrives early. This also surfaces load failures at start rather than on first dictation. Pre-touch/mmap the model so pages are resident.
- **[MED] `garbage-hallucinated-output`** (history) — Non-speech audio (keyboard clatter, fan, music, cough) produces a confident garbage/hallucinated transcript, or repetition-loop artifacts (TDT decoders can emit repeated tokens on noise).
  - **Handling:** Apply lightweight sanity guards before insert: drop results below a minimum signal threshold (use the recorder's RMS/VAD voiced-frame ratio — if almost no voiced audio, treat as empty) and detect pathological repetition (same token/word repeated beyond a threshold), trimming or discarding with a logged warning. Keep History storing the full unedited result + timestamp + audio-duration so the user can recover/inspect what was produced. Never auto-execute; insertion is plain text only.
- **[MED] `very-long-audio-very-long-text`** (recorder) — An extremely long recording (user forgets to stop, leaves it running for minutes) produces a very long transcript.
  - **Handling:** Enforce a hard max recording duration (configurable, e.g. 120s) with an auto-stop and a visible UI cue as it approaches the cap; reuse the existing duration signal that feeds the progress estimate. For long clips, chunk audio internally and stream-insert per chunk rather than one mega-paste. Cap memory by stopping capture at the limit instead of buffering indefinitely. Store the full text in History regardless of insert truncation.
- **[MED] `model-revision-drift-on-uv-bump`** (logging) — A `uv` python bump changes the interpreter path/identity, or an onnx-asr/onnxruntime upgrade changes expected model files or ABI, so a previously-good resident model no longer loads after an update.
  - **Handling:** Pin onnxruntime, onnx-asr, and the model repo revision together; record the model id+revision and onnxruntime version in the `model_ready` marker and re-validate on start, auto-triggering a re-resolve/repair if they drift. Treat a post-update load failure as a first-class `setup --repair` prompt rather than a silent dead daemon. Log the interpreter path, package versions, and model revision on every start for observability.
- **[MED] `engine-load-blocks-ui-thread`** (ui) — The 630MB model load / warmup is accidentally run on the Tk main thread (or the worker posts UI updates without marshalling), freezing the single-threaded Tkinter loop during the multi-second load.
  - **Handling:** Load, warmup, and recognize() all run exclusively on the engine worker thread/subprocess; the engine emits status/amplitude/duration events onto a thread-safe queue the Tk loop drains via after(). Never call Tk widget methods from the engine thread. This is the same marshalling discipline the recorder uses for amplitude — reuse it for engine progress.
- **[LOW] `non-english-when-language-en`** (config) — User speaks a non-English language (or code-switches) while the model is English-only Parakeet v2 with language=en.
  - **Handling:** Document clearly that v1 ships English-only (Parakeet v2) with language fixed to en. Don't pretend to detect language at runtime (adds latency/complexity). Make the model id swappable in config so a future multilingual model (e.g. parakeet-tdt-0.6b-v3) can be selected in `setup` without engine code changes. Surface the configured language in `status`/tray so expectations are set.
- **[LOW] `numeric-punctuation-formatting-surprises`** (engine) — Parakeet's built-in punctuation/capitalization/inverse-text-normalization formats unexpectedly: '20 26' vs 'twenty twenty-six', a forced trailing period, a capitalized first word on a fragment, spelled-out vs digit numbers, or it strips a literal spoken 'period'/'comma'.
  - **Handling:** Accept the model's native formatting as v1 behavior (auto-punctuation is a selling point) but make minimal post-processing pluggable at the engine output stage: optional toggles for strip-trailing-period and leading-capitalization, plus trim leading/trailing whitespace before insert. Keep raw output in History so users see exactly what the model produced. Don't hand-roll spoken-command parsing in v1 — document the formatting behavior so expectations match.


## 3. Clipboard injection & focus  (20 cases · 2 critical · 8 high)

- **[CRIT] `focus-changed-before-paste`** (injector) — User alt-tabs / clicks another window in the gap between record-stop and the synthesized paste, so the transcript lands in the wrong app.
  - **Handling:** Capture the focused-window identity at record-STOP (the moment transcription starts), not at paste time. Add `get_focused_window()`/`focus_token()` to the platform adapter (macOS: frontmost app PID/bundle id via NSWorkspace; Windows: GetForegroundWindow + GetWindowThreadProcessId). Before paste in the `inserting` state, re-read the focused window; if it changed, do NOT paste. Instead set the clipboard to the transcript, show a panel toast 'transcript copied — paste with Cmd/Ctrl+V' and add it to history. Make 're-focus check' configurable (strict/lenient) since some users want fire-and-forget.
- **[CRIT] `restore-too-soon-race`** (clipboard) — Clipboard is restored to its prior contents before the target app has actually read the pasted text, so the wrong (old) content gets pasted or paste is empty.
  - **Handling:** Do not rely solely on a fixed sleep before restore. Default the restore delay higher than the Windows reference's lower bound (use ~150ms baseline) and make it configurable. Prefer a confirmation-driven restore where possible: on Windows, defer restore until WM_CLIPBOARDUPDATE or a verified paste, or simply DELAY restore to a later safe point (e.g., several hundred ms, or on next idle). Better default: do NOT auto-restore at all by default — leave the transcript on the clipboard (most dictation tools do this); offer 'restore previous clipboard after paste' as an opt-in with a generous delay. This eliminates the entire race class for cautious users.
- **[HIGH] `no-editable-field-focused`** (injector) — No text input is focused at paste time (desktop, a button-only toolbar, a read-only view, a fullscreen game), so the paste is a silent no-op.
  - **Handling:** Treat paste as best-effort and NEVER silently drop the transcript. Always keep the transcript recoverable: leave it in history, and if paste cannot be verified, surface 'copied to clipboard — paste manually' in the panel. Cannot reliably detect 'is a text field focused' cross-platform, so rely on the post-paste verification fallback (see paste-failed-detection) rather than pre-checks.
- **[HIGH] `macos-secure-event-input`** (injector) — Target has EnableSecureEventInput active (password fields in browsers, iTerm2/kitty/Ghostty secure keyboard entry, login items), so synthetic Cmd+V is silently filtered.
  - **Handling:** Detect secure input state on macOS via `IsSecureEventInputEnabled()` (Carbon) before synthesizing the paste. If true, skip the synthetic chord, leave the transcript on the clipboard, and show 'secure input active — press Cmd+V to paste'. Log the event. Document that terminals with Secure Keyboard Entry and password fields require manual paste. Do not retry (retries also fail and risk multi-paste later).
- **[HIGH] `windows-uipi-elevated-target`** (injector) — Focused target is an elevated/admin window (UAC, elevated PowerShell, Task Manager) while yohoho runs unelevated; UIPI blocks the synthetic Ctrl+V from a lower-integrity process.
  - **Handling:** Accept that an unelevated daemon cannot inject into elevated windows (by design; do NOT request elevation — that's a security regression for a background daemon). Use the post-paste verification fallback: if clipboard contents were not consumed / focus is an elevated window we can't introspect, leave transcript on clipboard and toast 'elevated window — press Ctrl+V'. Document this limit.
- **[HIGH] `clipboard-open-access-denied`** (clipboard) — Another app or a clipboard manager holds the clipboard open at the instant yohoho tries to read (save) or write (set), causing OpenClipboard ACCESS_DENIED on Windows / pasteboard contention on macOS.
  - **Handling:** Wrap every clipboard open/read/write in a bounded retry: up to ~5 attempts with ~30-50ms backoff (the documented ~100ms-total mitigation). Make save, set, and restore each independently fault-tolerant. If SAVE fails, mark prior clipboard as 'unknown' and SKIP the restore step entirely (never write empty/garbage over the user's clipboard). If SET fails after retries, abort paste, keep transcript in history, and toast the failure. Log every retry/abort with which step failed.
- **[HIGH] `prior-clipboard-nontext-clobbered`** (clipboard) — User's prior clipboard held an image, file list (CF_HDROP / file URLs), rich text/HTML, or other non-plain-text format; save/restore only round-trips plain text and destroys it.
  - **Handling:** Save/restore must be FORMAT-AWARE, not text-only. On Windows enumerate all available clipboard formats and round-trip the raw data (or at minimum detect non-text and refuse to clobber-then-restore); on macOS save/restore all NSPasteboard types/items, not just NSString. If full-fidelity save is infeasible with pyperclip, detect that prior content is non-text and switch policy: do NOT restore (leave transcript), and warn 'previous clipboard was an image/file — left transcript on clipboard'. Never write a text-only 'restore' over non-text content.
- **[HIGH] `multi-paste-duplication-on-retry`** (controller) — A retry or double-fire of the paste chord inserts the transcript twice (or the user manually pastes after an auto-paste also succeeded late).
  - **Handling:** Make the `inserting` state strictly single-shot and idempotent: one paste attempt per transcription, guarded by the controller state machine (ignore re-entry while in `inserting`). Do NOT auto-retry the synthetic paste — instead fall back to 'leave on clipboard, prompt manual paste'. Debounce the hotkey so a held/repeated key cannot trigger a second stop→paste. Tag each transcription with an id so a late async paste of a stale clipboard can't be re-issued.
- **[HIGH] `cancel-midstream-clipboard-restore`** (controller) — User cancels/closes while dictating or mid-transcribe; the clipboard must be left untouched and no paste must fire, but a naive flow has already saved/overwritten the clipboard.
  - **Handling:** Defer ALL clipboard mutation to the last possible moment — only save+set inside the `inserting` state, AFTER recognize() returns and AFTER the abort check. A cancel transition from recording/transcribing must short-circuit to `idle` having touched nothing. Make `inserting` atomic and cancel-checked at entry. On daemon quit mid-insert, ensure the restore (if policy enabled) still runs via a finally/cleanup so the clipboard is never left in yohoho's transient state.
- **[HIGH] `paste-failed-detection-fallback`** (injector) — There is no single reliable signal that a synthetic paste succeeded; many of the above failures are silent, so the tool needs a uniform detection + fallback rather than per-case guessing.
  - **Handling:** Adopt a 'paste is best-effort, transcript is never lost' policy as the architectural invariant: (1) ALWAYS write the transcript to history before/independent of paste; (2) capture focus token at stop and verify it's unchanged before pasting; (3) on macOS check IsSecureEventInputEnabled and skip if set; (4) where feasible (Windows), after the paste delay, check whether the clipboard sequence/owner changed or whether the field consumed it, as a soft success signal; (5) on any non-confirmation, leave transcript on clipboard, show a panel state 'copied — paste manually (Cmd/Ctrl+V)', and log the reason. Never auto-retry the synthetic chord (prevents duplication). This single policy covers no-field, secure-input, UIPI, focus-change, IME, and async-app cases.
- **[MED] `clipboard-manager-history-pollution`** (clipboard) — A clipboard manager records yohoho's transient set + restore as two separate history entries, polluting the user's clipboard history and possibly capturing sensitive dictated text permanently.
  - **Handling:** Where supported, mark the clipboard write as transient/concealed so managers skip it: Windows — set the `ExcludeClipboardContentFromMonitorProcessing` / `CanIncludeInClipboardHistory` clipboard formats (and `CanUploadToCloudClipboard`=0) so Win+V/cloud history ignore it; macOS — there is no universal opt-out, so document it and rely on the no-restore default to at least avoid the double entry. Add a config note in setup. This also partially mitigates the privacy story for a 'fully local' product.
- **[MED] `very-large-transcript`** (clipboard) — A very long dictation (multi-minute monologue) produces a huge transcript; some apps truncate, choke, or paste slowly; clipboard set may hit size limits or be very slow.
  - **Handling:** Scale the pre-restore delay with transcript length (e.g., base 150ms + a few ms/KB, capped). Keep the full transcript in history regardless so truncation is recoverable. Consider a soft cap / warning for extreme lengths. Do not chunk-paste by default (reintroduces ordering/duplication risk); clipboard paste handles large text fine — the risk is timing, addressed by the adaptive delay.
- **[MED] `paste-into-password-manager-or-pw-field`** (injector) — Paste lands in a password manager's search/entry field or a website password field, putting dictated text where it shouldn't go — or, conversely, the transcript is sensitive and ends up in a place that logs it.
  - **Handling:** On macOS, the secure-input detection (case macos-secure-event-input) already blocks most password fields — lean on it. Do not special-case password managers beyond that. Pair with the transient/concealed clipboard flags so a transcript that does pass through is not retained by clipboard history. Document that yohoho is dictation, not a credential tool.
- **[MED] `panel-steals-focus-or-overlaps-paste`** (ui) — The always-on-top Tkinter panel (or pystray interaction) momentarily takes focus or the act of showing/hiding it changes the foreground window right as paste fires.
  - **Handling:** Ensure the panel is created non-activating / does-not-take-focus (macOS NSWindow nonactivating + canBecomeKey=NO; Windows WS_EX_NOACTIVATE) as already specified in the panel spec ('does-not-take-focus'). Explicitly EXCLUDE yohoho's own windows from the focused-window capture/verification so the tool never treats itself as the target. Verify the focus token ignores the daemon's own PID/bundle id.
- **[MED] `modifier-keys-physically-held-during-paste`** (injector) — The user is still physically holding a modifier (or the hotkey itself uses a modifier) when the synthetic paste chord is sent, producing a corrupted chord (e.g., Cmd+Shift+V, Ctrl+Alt+V) or a different command.
  - **Handling:** Before synthesizing the paste chord, normalize keyboard state: explicitly release any modifiers yohoho knows the hotkey used, and add a small settle delay after stop so the user can release keys (the recognize() latency usually covers this). Synthesize the paste as an isolated, fully-specified chord (press Cmd/Ctrl, press V, release V, release Cmd/Ctrl) and verify pynput modifier bookkeeping is reset afterward so no modifier sticks.
- **[MED] `rapid-successive-dictations-clipboard-overlap`** (controller) — User fires a second dictation before the first's paste+restore cycle has completed, so two inserting flows overlap and stomp each other's saved clipboard.
  - **Handling:** Serialize the entire inserting/restore cycle in the controller — a new recording cannot enter `inserting` until the previous restore (if any) has completed. Use a single owning lock around save/set/paste/restore. If a new dictation starts mid-cycle, cancel the pending restore of the previous one (the user is clearly continuing to work) rather than restoring stale content.
- **[MED] `logging-leaks-transcript-contents`** (logging) — The error-logging/observability layer logs full clipboard/transcript contents when diagnosing paste failures, persisting sensitive dictated text to disk in plaintext.
  - **Handling:** In the logging/observability layer, NEVER log transcript or clipboard payloads at default levels — log only metadata (length, format, which step failed, focus changed yes/no, secure-input yes/no, retry count). Gate any content logging behind an explicit opt-in debug flag with a clear warning. This keeps observability useful for the paste-failed-detection work without leaking content.
- **[LOW] `unicode-emoji-rtl-loss`** (injector) — Transcript contains emoji, non-BMP characters, combining marks, or RTL text (Arabic/Hebrew); clipboard encoding or paste mangles or drops them.
  - **Handling:** Always set clipboard as full Unicode: Windows use CF_UNICODETEXT explicitly (don't let pyperclip fall back to a narrow format); macOS NSPasteboard NSString is already UTF. Round-trip a unicode test string (with smart quotes/em-dash/emoji) in a setup self-test. Do not transform/normalize the transcript before paste beyond what's intended.
- **[LOW] `active-ime-composition`** (injector) — An IME (Japanese/Chinese/Korean, or any composition) is mid-composition in the target when the synthetic paste chord arrives; the paste interleaves with or is swallowed by the composition buffer.
  - **Handling:** Cannot reliably detect IME composition cross-platform from an external process; rely on post-paste verification + 'press Cmd/Ctrl+V to paste' fallback when paste isn't confirmed. Document the limitation. Avoid sending any keystrokes other than the single paste chord (no Escape/clear) which could disturb a composition.
- **[LOW] `transcript-empty-or-whitespace`** (controller) — recognize() returns empty/whitespace/just-punctuation (silence, noise, very short clip), but the inserting flow still clobbers the clipboard and pastes nothing meaningful.
  - **Handling:** Guard the `inserting` state on a non-empty, non-whitespace transcript: if empty after trimming, skip clipboard save/set/paste entirely, return to `idle`, optionally flash the panel ('no speech detected'), and do NOT add a blank history entry. Zero clipboard side effects when there's nothing to paste.


## 4. Background daemon & process lifecycle  (23 cases · 3 critical · 14 high)

- **[CRIT] `double-start-race`** (daemon) — User runs `yohoho start` twice (or autostart fires while a manual instance already runs), spawning two live daemons.
  - **Handling:** Make `start` acquire an exclusive OS-level advisory lock on `<datadir>/yohoho.lock` (fcntl.flock on macOS, msvcrt.locking on Windows) and HOLD the fd open for the daemon's whole life — not just check-then-write. If the lock is already held, print the running PID from the PID file and exit 0 (idempotent start). The lock is the source of truth; the PID file is informational. Acquire the lock in the parent before detaching so the CLI can report 'already running' synchronously.
- **[CRIT] `listener-thread-dies-silently`** (daemon) — The pynput keyboard.Listener thread throws and dies, but the daemon process stays alive.
  - **Handling:** Wrap the listener callback so exceptions are logged, never silently swallowed. Run a supervisor/heartbeat: the controller checks `listener.running`/`listener.is_alive()` on a timer (~every 2–5s); if dead, log it, and attempt a bounded auto-restart of the listener (re-create keyboard.Listener). If restart fails (e.g. permission revoked), surface it: update tray icon to a warning state and flash the Tk panel with 'hotkey inactive — check permissions'. Have `status` report listener health, not just process liveness.
- **[CRIT] `macos-permission-silent-revoke`** (permissions) — macOS TCC Input Monitoring or Accessibility is revoked while the daemon runs (OS update, user toggle, or uv python path change invalidating the granted identity).
  - **Handling:** On startup and on a periodic health check, actively probe permissions (AXIsProcessTrusted() for Accessibility; IOHIDCheckAccess(kIOHIDRequestTypeListenEvent) for Input Monitoring where available). Record the granted interpreter path; on `start`, if the current sys.executable differs from the recorded granted path, warn loudly and re-run the permission guidance. Surface all permission loss in tray + panel + log; `status` must show permission state on macOS.
- **[HIGH] `stale-pidfile-after-crash`** (daemon) — Daemon crashes or is SIGKILLed; the PID file remains on disk pointing at a dead (or reused) PID.
  - **Handling:** Never trust the PID number alone. Treat the held flock as primary liveness: if `start` can ACQUIRE the lock, any existing PID file is stale by definition — overwrite it. As a secondary guard, write a JSON PID file `{pid, start_time, exe_path, lock_inode}` and on `status` verify both `os.kill(pid,0)` AND that the process start-time/exe matches (psutil.Process(pid).create_time()/.exe()), so a recycled PID can't masquerade. Auto-remove a verified-stale PID file with a logged 'recovered from stale pidfile' event.
- **[HIGH] `stop-signal-then-hang`** (daemon) — `stop` sends SIGTERM but the daemon is wedged mid-transcription (the single blocking onnx-asr recognize() call) and doesn't exit.
  - **Handling:** Run the engine recognize() on a worker thread, keep the Python main thread in a signal-receptive loop (or use signal.set_wakeup_fd). On SIGTERM: set a shutdown Event, let the current recognize() finish if it's near done, but bound it — `stop` should send SIGTERM, wait up to ~5s polling the lock release, then escalate to SIGKILL with a warning. Document that an in-flight transcript may be dropped on forced stop. Flush history/logs in the SIGTERM handler before exiting.
- **[HIGH] `sigterm-graceful-cleanup`** (daemon) — Clean SIGTERM/SIGINT (Ctrl+C in foreground, `stop`, OS shutdown) must flush state and release resources in the right order.
  - **Handling:** Install one handler for SIGTERM and SIGINT (and on Windows handle CTRL_CLOSE_EVENT via win32api.SetConsoleCtrlHandler) that runs an idempotent shutdown sequence in order: (1) stop hotkey listener, (2) abort/stop sounddevice stream and close it, (3) release any held modifier keys via pynput, (4) restore the saved prior clipboard, (5) flush+fsync history and log files, (6) destroy the Tk panel via its queue, (7) release flock and unlink the PID file last. Guard the sequence so a second signal during shutdown doesn't re-enter it.
- **[HIGH] `sleep-wake-stale-mic`** (recorder) — Machine sleeps and wakes; the sounddevice/PortAudio mic stream and/or device index is now invalid.
  - **Handling:** Don't hold a single PortAudio stream open across the daemon's life — open the InputStream per recording session and close it on stop, so each recording re-resolves the device. Subscribe to wake notifications (macOS: NSWorkspace didWakeNotification via pyobjc; Windows: WM_POWERBROADCAST wake) and on wake re-query devices and re-arm the listener. Resolve the configured input by NAME, not index, with a fallback to the default device and a logged warning when the configured device is gone.
- **[HIGH] `sleep-wake-stale-listener`** (daemon) — After sleep/wake (or display reconfiguration / monitor unplug), the global hotkey event tap stops delivering events.
  - **Handling:** On the same wake notification, proactively tear down and recreate the keyboard.Listener (cheap, ~ms). On macOS specifically, treat a tap that stops producing events as 'disabled' — the heartbeat from listener-thread-dies-silently should detect 'no events + tap disabled' and re-create the listener. Log every re-arm with a reason ('rearmed listener after wake').
- **[HIGH] `crashed-but-tray-alive`** (daemon) — The daemon's worker logic dies but a child/tray process (pystray) or the autostart launcher still believes it's alive.
  - **Handling:** Keep the daemon single-process (controller owns the Tk loop, tray, listener, recorder) so there's exactly one liveness anchor — the held lock. Run an internal watchdog: each subsystem (listener, recorder, engine, ui-queue) updates a last-heartbeat timestamp; a monitor thread that sees a subsystem stall beyond a threshold logs it and either restarts that subsystem or, if unrecoverable, exits the whole process cleanly (releasing the lock) so autostart/tray reflect reality. Tray menu must read live controller state, not a cached boolean.
- **[HIGH] `autostart-before-session-ready`** (setup) — Autostart launches the daemon before the GUI session / display server / TCC is ready, so Tk or the listener fails to initialize.
  - **Handling:** Make startup resilient: retry Tk/display and listener init with backoff for up to ~30–60s, logging each attempt, instead of crashing. On macOS use a LaunchAgent (per-user, runs in the Aqua session) NOT a LaunchDaemon, and set it to relaunch (KeepAlive with crash-only semantics) so a too-early start gets retried. Defer creating the Tk panel until first hotkey press (lazy UI) so an unavailable display at login doesn't kill the process. Detect headless/no-session and idle-wait rather than exit.
- **[HIGH] `mic-held-blocking-other-apps`** (recorder) — The daemon holds the microphone open continuously, blocking or interfering with other apps (Zoom, Teams, OBS) that want exclusive/shared mic access.
  - **Handling:** Open the input stream ONLY during an active recording session (idle state holds no stream), and always use shared mode. This frees the mic between dictations and removes the always-on mic indicator (a brand/trust win for a privacy tool). If opening the stream fails because another app holds it, catch the PortAudio error, log it, and flash the panel with 'microphone unavailable (in use by another app)' instead of failing silently.
- **[HIGH] `cancel-midstream-state`** (controller) — User cancels/closes the panel while dictating (mid-record or mid-transcribe), and the daemon must abort cleanly without leftover state.
  - **Handling:** Add an explicit CANCEL transition from both recording and transcribing back to idle. Cancel from recording: stop+close the stream, discard the buffer, do NOT call the engine, return to idle. Cancel from transcribing: set a 'cancelled' flag the engine thread checks immediately after recognize() returns — if set, drop the result (no clipboard write, no paste) and return to idle, AND skip writing it to history. Ensure cancel is idempotent and that the controller resets to a known-good idle on any cancel.
- **[HIGH] `update-while-running`** (setup) — User runs `uv tool upgrade yohoho` (or npm reinstall) while a daemon instance is running.
  - **Handling:** Have the CLI detect a version mismatch between the running daemon (record its version in the PID/state file) and the installed package; on `start`/`status`/`stop`, if they differ, advise 'a newer yohoho is installed — run `yohoho restart`'. Provide `yohoho restart` that does a clean stop (graceful) then start. Version the lock/PID/config file formats and migrate or refuse with a clear message rather than misbehaving. After any upgrade, re-check the macOS interpreter-identity (see macos-permission-silent-revoke) since the grant may need redoing.
- **[HIGH] `datadir-unwritable-lock`** (logging) — The per-user data dir is missing, read-only, on a full disk, or on a network/cloud-synced path, so the lock/PID/history/log can't be written.
  - **Handling:** On `start`, create the data dir and assert it's local + writable (write a temp probe file, reject paths under known cloud-sync roots with a clear message). If the lock cannot be acquired due to an IO/permission error (vs. genuinely held), FAIL CLOSED — do not start unlocked; print the exact path and error. Keep the log destination separate and fall back to stderr/syslog if the file log can't be opened, so logging never silently dies.
- **[HIGH] `engine-load-failure-startup`** (engine) — The daemon starts but the engine fails to load (model missing/corrupt, onnxruntime native lib fails, OOM loading the 660MB int8 model).
  - **Handling:** Load the model lazily/guarded: catch load failures, log the full error with the model path and a remediation hint ('run `yohoho setup` to re-download the model'), and put the controller into a clearly-degraded state (panel shows 'model not loaded') rather than crashing the daemon or silently no-op'ing. Validate model file size/hash on startup; if corrupt, surface it. Keep `import onnxruntime` ordering discipline noted in the handoff to avoid native-lib init failures.
- **[HIGH] `stuck-modifier-on-crash`** (injector) — Daemon crashes during the synthetic paste chord, leaving Cmd/Ctrl (or V) logically held down system-wide.
  - **Handling:** Send paste key-down and key-up via a try/finally so the release always fires even on exception. In the SIGTERM/crash handler and on daemon startup, defensively release the known modifier set (keyUp for Cmd, Ctrl, Shift, Alt, V) via the platform injector so a fresh start also clears a previously-stuck modifier. Bound the down→up sleep and never block indefinitely between them.
- **[HIGH] `no-observability-into-lifecycle`** (logging) — Lifecycle events (start, crash, listener restart, permission loss, wake re-arm, stale-pid recovery) aren't logged, so users/maintainers can't diagnose 'it stopped working'.
  - **Handling:** Add a rotating lifecycle log at `<datadir>/logs/yohoho.log` (RotatingFileHandler, capped size) that records: process start/stop with version + interpreter path + PID, every state transition, listener/recorder/engine health-check results and restarts, permission probe results, sleep/wake and session events, stale-pid recoveries, and every caught exception with traceback. `status` should print the last N lifecycle events and the current health of each subsystem, not just 'running'. Never log transcript content to the lifecycle log (privacy) — history is separate and opt-out-able.
- **[MED] `stop-status-not-running`** (daemon) — User runs `stop` or `status` when no daemon is running.
  - **Handling:** `stop` when nothing holds the lock: print 'yohoho is not running', clean up any stray PID file, exit 0 (idempotent stop — important for uninstall/update scripts). `status` must have three explicit states: running (lock held + identity verified), stopped (lock free, no/clean PID file), and crashed/stale (PID file present but lock free or identity mismatch) — print the recovery hint for the third.
- **[MED] `logout-fast-user-switch`** (daemon) — User logs out / locks / fast-user-switches; the per-user daemon's mic, listener, and clipboard context change underneath it.
  - **Handling:** Scope the lock/PID/data dir per-user (already implied by %LOCALAPPDATA% / ~/Library/Application Support) so each user gets an isolated instance — verify the lock path includes the user. Subscribe to session lock/switch notifications (macOS: NSWorkspace sessionDidResignActive/DidBecomeActive; Windows: WTS_SESSION_LOCK/UNLOCK via WM_WTSSESSION_CHANGE) and on resign-active: abort any in-flight recording, close the mic stream, and pause the listener; resume on become-active. Never hold the mic while the session is inactive.
- **[MED] `orphaned-after-parent-exit`** (daemon) — The detached daemon is orphaned or zombied because the double-fork/detach was done incorrectly.
  - **Handling:** On POSIX, detach properly: fork, setsid, fork again, redirect stdio to the log file, ignore SIGHUP; on Windows use DETACHED_PROCESS/CREATE_NO_WINDOW. Prefer registering a LaunchAgent (macOS) / scheduled-task-or-Startup (Windows) as the canonical run mechanism so the OS owns lifecycle and TCC identity is stable, with `start` delegating to it. Verify after spawn that the child acquired the lock (parent reads the new PID file) before reporting success.
- **[MED] `two-cli-commands-concurrent`** (daemon) — Two CLI commands race: e.g. `start` and `start` simultaneously, or `stop` while `start` is mid-detach.
  - **Handling:** Use the atomic flock acquisition as the single arbiter — the loser of the race can't get the lock and exits idempotently, eliminating the check-then-act TOCTOU. Install signal handlers as the very first thing in the daemon (before loading the 660MB model) so an early `stop` is honored; until handlers are installed, write a 'starting' state to the PID file so `stop` can wait-and-retry rather than declare 'not running'.
- **[MED] `no-graceful-on-os-shutdown`** (history) — OS shutdown/restart sends SIGTERM with a short grace window; flushing history/logs and restoring clipboard may not finish.
  - **Handling:** Keep history/log writes append-and-fsync per transcript (don't batch in memory) so there's nothing to flush at shutdown — the latest committed transcript is already durable. Make the SIGTERM handler do the minimum critical work first (release modifiers, restore clipboard) before best-effort flush, so even a truncated shutdown restores clipboard. Test under the OS shutdown path, not just `stop`.
- **[MED] `restart-command-missing`** (cli) — There is no first-class restart path, so recovery from a wedged/degraded daemon requires the user to know stop+start and to clean up stale files manually.
  - **Handling:** Ship `yohoho restart` = graceful stop (with the SIGTERM→wait→SIGKILL escalation) + cleanup of verified-stale lock/PID + start. Make `start` self-heal: if the lock is free but a PID file/log indicates an unclean prior exit, log 'recovered from unclean shutdown', clear stale state, and proceed. Document `restart` prominently as the recovery action in `status` output when a degraded/stale state is detected.


## 5. macOS permissions & bootstrap/install  (19 cases · 2 critical · 9 high)

- **[CRIT] `tcc-revoked-while-running`** (permissions) — User grants Input Monitoring + Accessibility at setup, then later revokes one (or macOS resets it after an OS update) while the daemon keeps running.
  - **Handling:** Treat TCC as runtime-volatile, not setup-once. PermissionsManager.check() must be callable live and cheap: (a) Accessibility via AXIsProcessTrusted() through pyobjc/ctypes; (b) Input Monitoring via IOHIDCheckAccess(kIOHIDRequestTypeListenEvent). Controller polls permissions on each idle->recording transition AND on a low-frequency timer (~5s) while running. On loss, enter a 'permission_lost' state: dot-panel shows a clear cyan message ('input monitoring off - click to fix'), tray icon flips to an alert glyph, and a structured PERMISSION_REVOKED event is logged. Also detect the no-op paste case: if recognize() produced text but no key events were delivered, surface 'paste blocked - check Accessibility'.
- **[CRIT] `uv-python-bump-new-tcc-identity`** (permissions) — A uv python patch bump (e.g. 3.11.9 -> 3.11.10) relocates the real interpreter binary to a new versioned path, so the TCC grant (bound to the old resolved binary) no longer applies to the now-running interpreter.
  - **Handling:** Pin the interpreter identity. (1) At setup, record the resolved real path of the python binary that received the grant (resolve the ~/.local/share/uv/python/cpython-3.11-... symlink to its patch dir) into config as `granted_interpreter_path`. (2) On every start/status, compare the current resolved interpreter path to the recorded one; on mismatch, immediately flag 'python was upgraded - permissions must be re-granted' and run the repair grant flow, never start silently. (3) Prefer a STABLE launch path: install a small launcher (LaunchAgent ProgramArguments / wrapper in ~/.local/bin) whose binary path never changes. (4) Document for Phase 2 that a signed .app bundle with a stable bundle identifier is the real fix — TCC then keys on the signing identity and survives interpreter moves.
- **[HIGH] `tcc-no-prompt-on-revoke`** (permissions) — After a revoke, the event tap silently returns no events and never re-prompts, so the user has no path back without manual System Settings navigation.
  - **Handling:** Never rely on an automatic re-prompt. When check() reports denied, the UI/tray action deep-links straight to the exact pane: open 'x-apple.systempreferences:com.apple.preference.security?Privacy_ListenEvent' and '...?Privacy_Accessibility' via `open`. `yohoho status` prints these URLs plus the exact interpreter path to add. `yohoho setup --repair` re-runs the guided grant flow and, where possible, calls AXIsProcessTrustedWithOptions({kAXTrustedCheckOptionPrompt: true}) to force the Accessibility prompt.
- **[HIGH] `tahoe-synthetic-event-filtering`** (injector) — macOS Tahoe filters/ignores synthetic key events posted by an unsigned interpreter, so the Cmd+V paste chord is silently dropped even with Accessibility granted.
  - **Handling:** (1) Make clipboard restore conditional/safe: when paste likely failed (Tahoe + unsigned + no observable effect) SKIP the restore so the transcript stays pasteable manually; log a 'synthetic_paste_filtered' event and toast 'press Cmd+V to paste'. (2) Add a config fallback insert mode that leaves text on the clipboard and notifies rather than synth-pasting. (3) Detect OS build at setup (sw_vers) and on Tahoe warn that an unsigned interpreter may have paste filtered, recommending the signed-binary path. (4) Phase-2 fix: ship a codesigned + notarized launcher/app so synthetic events from a trusted signed identity are honored.
- **[HIGH] `node-missing-or-wrong-version`** (setup) — `npm i -g yohoho` runs under a Node too old for the shim, or the user has no Node at all but tries to install another way.
  - **Handling:** Declare a strict `engines.node` (>=18) in package.json so npm warns/blocks. The shim's very first line checks `process.version` and exits with a crisp message + the minimum version BEFORE doing anything else. The shim does as little as possible itself (bootstrap uv + delegate); all real logic lives in python. `yohoho status` (and a `yohoho doctor`) report node version, uv presence/version, python version, and the install-state machine so a half state is diagnosable.
- **[HIGH] `uv-install-blocked-av-or-policy`** (setup) — The curl/irm uv installer is blocked by corporate endpoint protection / MDM, or (on the Windows side of the cross-platform shim) by PowerShell ExecutionPolicy, so uv never lands.
  - **Handling:** Wrap the uv install with explicit detection and messaging: if the installer exits non-zero, print a diagnosis ('uv could not be installed - likely AV/MDM or ExecutionPolicy') with manual remediation (Homebrew `brew install uv` on mac; `powershell -ExecutionPolicy Bypass` guidance on Windows; detect a pre-installed uv so corp users can pre-provision it). Always check `command -v uv` first and skip install if present. Use `-ExecutionPolicy Bypass -Scope Process` in the invocation we control rather than asking users to change machine policy.
- **[HIGH] `network-loss-mid-model-download`** (setup) — Network drops (or HF rate-limits) partway through the 660MB model download during first `setup`.
  - **Handling:** Make model acquisition resumable and verified: rely on huggingface_hub resumable downloads, then verify integrity (expected files present + size/etag/sha against a pinned manifest) before marking setup complete. Write a `model_ready: true` flag to the data dir ONLY after verification. `setup` must be re-runnable and resume; `status` must report model state ('model: incomplete - run yohoho setup'). On engine load failure, detect a bad cache and offer `yohoho setup --redownload` rather than surfacing a raw stack trace.
- **[HIGH] `uv-bin-not-on-path`** (setup) — uv installs the `yohoho` shim into ~/.local/bin which is not on the user's PATH, so `yohoho` is 'not found' immediately after a successful install.
  - **Handling:** After uv tool install, run `uv tool update-shell` to add ~/.local/bin to PATH in the shell rc, then detect whether the shim is resolvable in the CURRENT shell. Since the running shell will not pick up the new PATH until restart, print an explicit next step ('open a new terminal, or run: export PATH="$HOME/.local/bin:$PATH"') and show the absolute shim path. Better: have the npm shim (already on PATH via npm's global bin) remain the durable entrypoint that locates and execs the uv-installed python tool by absolute path, so `yohoho` works even when ~/.local/bin is not on PATH.
- **[HIGH] `partial-failed-install-half-state`** (setup) — Any step (uv, python pkg, model, permissions) fails midway, leaving an inconsistent mix of installed/uninstalled components with no single source of truth.
  - **Handling:** Define an explicit install-state machine persisted in the data dir (install_state.json with booleans/timestamps: uv_ok, pkg_ok, model_ok, perms_input_ok, perms_ax_ok, granted_interpreter_path, shim_version, pkg_version). Every step writes its flag only on verified success. `setup` is idempotent and resumes from the first unsatisfied flag; `status`/`doctor` render the full table with a clear next action per failed row. Provide `yohoho reset` / `--clean` to wipe state and start fresh deterministically.
- **[HIGH] `input-monitoring-vs-accessibility-partial-grant`** (permissions) — User grants only ONE of the two required TCC permissions (Accessibility but not Input Monitoring, or the reverse), because the two prompts/panes are separate and easy to half-complete.
  - **Handling:** Check and report the two permissions INDEPENDENTLY (IOHIDCheckAccess for Input Monitoring, AXIsProcessTrusted for Accessibility) everywhere — in `setup`, `status`, and at runtime. The setup flow verifies each grant individually with its own deep-link and a 'confirm granted' re-check loop before proceeding, and `status` shows two distinct rows ('input monitoring: granted', 'accessibility: DENIED -> fix') so the user knows exactly which pane to open.
- **[HIGH] `first-launch-prompt-needs-restart`** (permissions) — On the very first hotkey/paste attempt macOS shows the TCC prompt, but the grant only takes effect for a NEW process — the currently-running daemon (already denied) will not see the new permission until restart.
  - **Handling:** Drive permission grants during `setup` (before the long-running daemon is the thing that needs them), and after a grant is detected, automatically restart the daemon/listener so the new TCC state takes effect. After any grant, `status` says 'permissions granted - restarting yohoho to apply'. Do not ask the user to manually restart; the controller re-inits the platform hotkey listener once check() flips to granted.
- **[MED] `network-loss-mid-uv-or-pip`** (setup) — Network loss during `uv tool install` of the python package itself (not the model) leaves a partially-resolved environment.
  - **Handling:** uv tool install is largely atomic per environment, but treat any non-zero exit as 'install not complete': do not write the 'installed' state flag until a post-install self-check (`python -c 'import onnx_asr, onnxruntime, sounddevice, pynput'`) passes. `yohoho doctor` runs this import smoke test and reports the first missing dep. setup/install must be idempotent: re-running cleanly repairs (`uv tool install --reinstall yohoho`).
- **[MED] `corporate-proxy-airgapped`** (setup) — User is behind an authenticating corporate proxy or fully air-gapped; uv and HuggingFace downloads cannot reach the internet directly.
  - **Handling:** Respect standard proxy env vars (HTTP_PROXY/HTTPS_PROXY/NO_PROXY) and pass them through the shim to uv and huggingface_hub; document setting them. Support HF_ENDPOINT / a custom mirror and a fully offline path: a documented `yohoho setup --model-path /path/to/local/model` (or honoring a pre-populated HF cache / HF_HUB_OFFLINE=1) so air-gapped users can side-load the model and skip network steps. `status` surfaces 'offline mode / local model' as an explicit, supported state.
- **[MED] `config-corrupt-or-schema-mismatch`** (config) — config file is corrupted (truncated by a crash) or carries an old schema after an upgrade adds/renames keys (e.g. activation_key format, new permission fields).
  - **Handling:** Version the config (`schema_version`) and run a migration on load: known old->new migrations bump fields and rewrite atomically; unknown/garbled config => back it up to config.bak and regenerate defaults rather than crashing, logging a CONFIG_RESET event and notifying the user that their hotkey may need reconfiguring (`yohoho setup`). Write config atomically (temp file + os.replace) to avoid truncation. `doctor` validates config against the schema and reports drift.
- **[MED] `shim-pkg-version-drift`** (setup) — The npm shim version and the uv-installed python package version diverge (user updates one but not the other; or npm cache serves a stale shim).
  - **Handling:** Establish a compatibility contract: the shim reports its version and asks python for its version on startup; if major/minor are incompatible, print a precise 'shim X vs package Y mismatch - run: npm i -g yohoho@latest && uv tool upgrade yohoho'. Keep the shim->python interface narrow and backward-tolerant (ignore-unknown args on the python side; stable status JSON with its own version field). On first run after an npm upgrade, the shim ensures the matching python package version is installed (drive `uv tool install yohoho==<pinned>`).
- **[MED] `model-cache-deleted-under-running-daemon`** (engine) — The HF model cache is deleted (cleanup tool, user freeing space, or a synced home) while the daemon is running with the 660MB model resident.
  - **Handling:** On daemon start, verify model files exist + valid before declaring ready; if missing, auto-trigger a resumable re-download with a clear 'model cache missing, re-downloading 660MB' message and a UI/tray indicator rather than crashing. Prefer loading weights fully into RAM at startup (model resident by design) so a mid-session cache delete cannot fault the running process. `status` reports model-on-disk presence distinctly from model-loaded-in-memory.
- **[MED] `insufficient-disk-for-model`** (setup) — Less free disk than the 660MB model (plus temp/extraction overhead) during first download.
  - **Handling:** Pre-flight a free-space check in `setup` before downloading: require headroom (~1.5GB to cover the model + temp) and abort early with 'need ~1.5GB free, have X' if short. After download, verify final size/integrity. On a mid-download ENOSPC, fail cleanly, delete the partial, and report the disk shortfall explicitly. `status`/`doctor` shows model size vs free disk.
- **[MED] `second-machine-synced-home-stale-grant`** (permissions) — A user's home dir (config + recorded granted_interpreter_path + maybe ~/.local) is synced/migrated to a second Mac (Migration Assistant, dotfile sync), but TCC grants do NOT transfer.
  - **Handling:** Never trust persisted 'permissions granted' flags as authoritative — always live-check TCC at runtime (see tcc-revoked-while-running) and verify granted_interpreter_path actually exists AND matches the current resolved interpreter. Bind machine-specific state (interpreter path, perms flags) to a host/arch key so a synced config from another machine is recognized as 'not set up here' and triggers `setup` on first run with a clear 'new machine detected - grant permissions' message.
- **[LOW] `multi-user-same-mac`** (daemon) — Two macOS user accounts on one machine both install/run yohoho; TCC and the model cache are per-user but installs may collide or one global install serves both.
  - **Handling:** Keep all config/data/state strictly per-user under ~/Library/Application Support/yohoho and per-user HF cache, and run permission checks per-user at runtime so each account is guided through its own grant. Use a per-user single-instance lock (lockfile/launchagent label) so one user cannot spawn duplicate daemons; document that each macOS user must run `yohoho setup` once. Do not assume a global install implies a configured user.


## 6. History & error-logging  (24 cases · 3 critical · 6 high)

- **[CRIT] `transcript-text-leaks-into-log-at-default-level`** (logging) — A debug/info log statement accidentally includes the recognized transcript string (or a clipboard payload, or an exception that stringifies the text), leaking the user's private speech into the rotating log file.
  - **Handling:** Hard architectural rule: transcripts NEVER enter the logging subsystem. Enforce it, don't just document it: (1) route all transcript text through a single TranscriptText wrapper type whose __repr__/__str__ returns '<transcript len=NN sha=xxxx>' so even an accidental f-string logs a redaction not the words; (2) add a logging.Filter installed on every handler that scrubs known-sensitive fields; (3) only log transcript metadata (length, word count, duration, latency); (4) a real transcript may be logged ONLY behind an explicit, off-by-default `--debug-transcripts` opt-in that prints a loud one-time warning. Add a unit test that asserts no log line equals/contains a fixture transcript.
- **[CRIT] `history-opt-out-must-be-honored-everywhere`** (history) — User disables history (privacy preference / corporate policy), but some code path (the 're-paste last' feature, a debug log, or a crash dump) still persists the transcript text.
  - **Handling:** Single gate: a HistoryStore.enabled flag checked at the one write site; when off, NOTHING transcript-related touches disk. For 're-paste last', when history is off keep the last transcript ONLY in daemon memory (lost on restart) and never persist it — document that re-paste-last requires the daemon to stay alive. Make opt-out a first-class setup question and reflect the current state in `yohoho status` ('history: off') so the user can verify. Provide `yohoho history clear` that also wipes the in-memory last-transcript.
- **[CRIT] `user-sees-no-error-because-panel-already-hidden`** (ui) — An error occurs at a moment the transient panel has already eased to 100% and hidden (or never showed), so the failure is completely invisible — the user just sees no text appear and has no idea why.
  - **Handling:** Make the panel a first-class error channel: add an explicit ERROR state to the controller state machine that the UI renders as a red-tinted dot-matrix flash with a short code/glyph (e.g. 'PERM' / 'PASTE' / 'MODEL'), holds ~2-3s (longer than the success finish), and does NOT auto-hide as fast. Because Tk is single-threaded, push this via the existing queue+after() marshal used for status/amplitude — add an error event type alongside recording/transcribing/idle. Pair every panel error with a persisted last_error so the user can also retrieve it via `yohoho status`. For errors with no active panel, force-show the panel briefly to surface the flash.
- **[HIGH] `history-file-readable-by-other-users`** (history) — The history file (and logs) are created with default umask permissions, so on a shared/multi-user machine other local accounts can read the user's entire dictation history.
  - **Handling:** Create the data dir and all sensitive files with restrictive perms at creation: on POSIX, os.open(path, O_CREAT|O_WRONLY, 0o600) and chmod the directory to 0o700; verify/repair perms on every startup. On Windows, the per-user %LOCALAPPDATA% already has per-user ACLs, but assert the dir resolves under the real user profile and isn't a redirected/synced share. Document the exact file locations and perms in `yohoho status`.
- **[HIGH] `history-on-cloud-synced-folder`** (history) — The per-user data dir resolves into a cloud-synced location (OneDrive Known Folder redirection on Windows, or iCloud Drive / Dropbox-managed Library on macOS), so private transcripts silently sync to the cloud despite the never-synced promise.
  - **Handling:** Pin storage to known-local roots: Windows %LOCALAPPDATA% (Local, which OneDrive does NOT sync) not %APPDATA% (Roaming, which can); macOS ~/Library/Application Support/yohoho. At startup, detect if the resolved data dir path contains a known sync provider marker (OneDrive/Dropbox/iCloud path segments or reparse points) and surface a one-time WARNING via `yohoho status` and a panel error-flash, offering to relocate. Drop a `.nosync`/exclude hint where supported.
- **[HIGH] `concurrent-write-two-daemons`** (history) — Two yohoho daemon instances are running (user ran `start` twice, or autostart launched one and the user launched another), and both append to history.jsonl / write the same rotating log concurrently, interleaving or corrupting records.
  - **Handling:** Enforce single-instance via a pidfile+OS lock in the data dir (the same lock that should gate `start`). For history writes specifically, even single-instance, do atomic appends: open in append mode and write one os.write() of a single line terminated by \n (POSIX append to a file opened O_APPEND under PIPE_BUF is atomic; keep lines well under 4KB or take an advisory lock). On Windows use msvcrt.locking / a mutex around the append. Never read-modify-write the whole file to append.
- **[HIGH] `history-clear-not-actually-gone`** (history) — `yohoho history clear` truncates/deletes the JSONL but the transcript data survives elsewhere: in the rotating logs, in OS backups (Time Machine / File History / VSS shadow copies), in the in-memory last-transcript buffer, or recoverable from the freed disk blocks.
  - **Handling:** Make 'clear' comprehensive and honest: delete/overwrite history.jsonl AND any cached last-transcript AND assert logs never contained text (which the no-leak rule guarantees). Be explicit in the command output about what clear does and does NOT cover (OS-level backups/snapshots are outside the tool's reach) — don't over-promise secure erase. Offer `clear` to also flush the in-memory buffer. Consider best-effort overwrite-then-unlink, while documenting that on SSDs/COW filesystems this is not guaranteed.
- **[HIGH] `status-command-no-last-error-surfaced`** (ui) — `yohoho status` reports running/stopped but gives no recent-error or health summary, so a user whose pastes silently fail has no CLI way to diagnose without reading raw logs.
  - **Handling:** `yohoho status` must show: running/stopped+pid+uptime; model loaded yes/no; permission states (Input Monitoring / Accessibility on macOS) live-checked; last N errors with timestamps and human messages; data dir path; history on/off + entry count + size; log file path + current size. Persist a small last_error.json (or read the tail of the log) so status works even after the panel is gone. Make `status` the primary self-service diagnostic, and have panel error-flashes literally tell the user 'run yohoho status'.
- **[HIGH] `uncaught-exception-vanishes-in-daemon`** (logging) — The daemon runs detached with no console attached, so an uncaught exception (or a fatal native crash in onnxruntime/sounddevice) dies silently with the traceback going to a closed stdout/stderr — nothing logged, panel just disappears.
  - **Handling:** Install a global crash net before going detached: sys.excepthook + threading.excepthook to log full tracebacks to the rotating log; faulthandler.enable() writing native fatal tracebacks to a dedicated crash file (faulthandler can dump on SIGSEGV/SIGABRT); on macOS/Linux also redirect the daemon's stdout/stderr to a startup.log so import-time and native loader failures are captured. Write a 'clean shutdown' marker on graceful exit so the NEXT start can detect a prior unclean crash and surface 'yohoho recovered from a crash — see logs' in status/panel.
- **[MED] `jsonl-partial-line-on-crash`** (history) — The daemon is killed (crash, power loss, `stop` mid-write) while a history line is half-written, leaving a truncated final line that breaks naive json.loads-per-line parsing in `yohoho history`.
  - **Handling:** Writer side: write the full line in one syscall and fsync (or at least flush) after each record so a record is either fully present or absent; never leave a buffered tail. Reader side: parse line-by-line, skip+count malformed lines instead of aborting, and on a malformed FINAL line just ignore it. Provide `yohoho history --repair` that rewrites the file dropping unparseable lines (atomic temp-file + rename). Treat a trailing partial line as expected, not exceptional.
- **[MED] `history-unbounded-growth`** (history) — A heavy daily dictation user accumulates history.jsonl forever; over months it reaches hundreds of MB, slowing `yohoho history`, bloating the data dir, and increasing the blast radius if the file leaks.
  - **Handling:** Cap by BOTH count and age with sane privacy-forward defaults (e.g. keep last 1000 entries OR 30 days, whichever is smaller), configurable, with an explicit 'unlimited' opt-in. Enforce the cap by periodic compaction: write surviving lines to history.jsonl.tmp then atomic rename (cheap because JSONL). Run compaction on daemon start and after every N appends, never on the hot dictation path. Surface current size/oldest entry in `yohoho status`.
- **[MED] `target-app-capture-is-extra-pii`** (history) — Storing the target/focused app name (or worse, window title) per history entry — which the prompt floats as a maybe — records sensitive context: window titles often contain document names, recipient names, URLs, ticket numbers.
  - **Handling:** If captured at all, store ONLY a coarse, non-sensitive app identifier (bundle id like com.apple.Terminal / process exe basename), NEVER the window title or document name. Make app-capture a separate opt-in defaulting OFF. Note that fetching even the app id on macOS may require Accessibility (already granted) but window titles require Screen Recording permission — another reason to avoid titles entirely.
- **[MED] `no-logs-command-user-cant-find-logs`** (logging) — Logs exist in a per-user data dir the user can't easily find (especially macOS ~/Library which is hidden by default), and there's no `yohoho logs` command, so bug reports come with no logs attached.
  - **Handling:** Add `yohoho logs` (print path + tail, with --follow to stream and --open to reveal in Finder/Explorer) and `yohoho logs --path`. Print the log path on every daemon start and in `status`. For bug reports add `yohoho doctor`/`yohoho diagnostics` that bundles redacted logs + config (with secrets/text stripped) + permission/model/version state into a single shareable file — and REMIND the user it contains no transcript text by design, so they can share it safely.
- **[MED] `log-rotation-unbounded-or-races`** (logging) — Logs grow without bound (no rotation), or RotatingFileHandler's rollover races/fails on Windows because another handle (a second instance, or a tailing `yohoho logs --follow`) holds the file open, raising on rotate or losing lines.
  - **Handling:** Use a size-capped RotatingFileHandler with a hard total budget (e.g. 1MB x 5 = 5MB) created once in the single daemon instance only. Guard rotation against Windows sharing violations (delay=True, and don't open the same file from a second process — `yohoho logs` should open read-only/shared, not via the logging handler). Surface total log footprint in status. Default level INFO; never DEBUG in production unless `--verbose`. Ensure logging failures can never take down the daemon (logging.raiseExceptions handling / try-guarded handler).
- **[MED] `disk-full-during-write`** (history) — The disk (or the user's quota) is full when the daemon tries to append a history line or write a log, raising ENOSPC on the hot dictation path.
  - **Handling:** Treat history and log writes as best-effort and fully isolated from the dictation path: wrap both in try/except, on failure degrade gracefully (skip the history append, set a last_error 'disk full — history not saved', flash the panel once) but STILL deliver the paste. Never let a logging/history I/O error propagate into the controller state machine. Detect low disk space at startup and warn.
- **[MED] `repaste-last-stale-or-empty`** (history) — 'Re-paste last transcript' is invoked when there is no last transcript (fresh daemon start with history off, or after `clear`), or it re-pastes a stale/wrong transcript because last-pointer wasn't updated after a cancel/empty recognition.
  - **Handling:** Define last-transcript semantics precisely: only successful, non-empty insertions update the last pointer; cancels and empty recognitions do NOT. Re-paste with no last available must give clear feedback (panel flash 'NO LAST' + status message), never paste stale data. Re-paste should reuse the SAME clipboard save/set/paste/restore dance as normal injection. When history is off, last lives in memory only and is explicitly absent after restart.
- **[MED] `cancel-midstream-must-not-persist-or-leak`** (history) — User cancels/closes a dictation mid-stream; the system must ensure no partial/abandoned transcript gets written to history, no last-transcript pointer is set, and the cancel event is logged without any audio/text content.
  - **Handling:** Cancel transitions the state machine to idle WITHOUT invoking the history write or last-pointer update; if recognize() already returned and the writer is mid-append, let that single atomic append complete but mark it (don't half-write). Log cancels as a metadata-only lifecycle event ('dictation cancelled at state=recording, dur=Ns'), never the partial text. This is the intersection with the cancel feature — keep the history/last invariants centralized so cancel can't bypass them.
- **[MED] `history-unicode-and-control-chars`** (history) — A transcript contains characters that break naive JSONL or terminal rendering: embedded newlines (multi-line dictation), emoji/CJK, NUL or other control chars, or extremely long single utterances.
  - **Handling:** Always serialize with json.dumps(ensure_ascii=False) which escapes newlines/controls into a single line — guaranteeing one record per line regardless of content. On the reader, when printing to a terminal, sanitize/escape control sequences (don't pass raw bytes that could contain ANSI). Enforce a max stored length per record (truncate with an ellipsis marker) to bound line size and keep appends atomic under PIPE_BUF.
- **[MED] `data-dir-uninitialized-or-unwritable`** (logging) — The per-user data dir doesn't exist yet, or exists but isn't writable (locked-down corporate machine, read-only profile, wrong owner after a sudo install), so the first history/log write fails at the worst time.
  - **Handling:** Bootstrap the data dir defensively at startup BEFORE configuring file logging: create dir tree with 0o700, verify writability with a probe write, and if it fails fall back to logging to stderr/console and a temp dir, then surface the failure loudly via the CLI exit code and message (since the daemon can't start). Never assume the dir exists. Check ownership matches the running user and warn on mismatch (the sudo-install footgun).
- **[MED] `tk-error-state-marshaling-race`** (logging) — An error originates on the audio thread or the engine thread and tries to drive the panel's error-flash directly, violating Tk's single-thread rule and either crashing Tk or silently dropping the error display.
  - **Handling:** Route ALL error UI through the same queue+after() marshal already mandated for status/amplitude/duration events — add a typed ('error', code, message) event. No background thread ever touches a Tk widget. The persisted last_error write (plain file I/O) is fine off-thread, but the panel flash must be enqueued. Unit-test that an error raised on a worker thread results in an enqueued error event, not a direct widget call.
- **[LOW] `history-write-on-hot-path-adds-latency`** (history) — Writing+fsyncing the history record (and verbose logging) happens synchronously between recognize() returning and the paste firing, adding latency or jank to the core 'text appears in ~1-2s' promise.
  - **Handling:** Order operations so the user-visible paste happens FIRST, then history/log writes happen after (or on a background writer thread/queue). Never run compaction or fsync on the path between recognize() and inject(). If using a writer thread, bound its queue and drop-with-warning rather than block. Keep logging at INFO and cheap on the hot path.
- **[LOW] `log-and-history-clock-skew-nonmonotonic`** (history) — Timestamps use wall-clock time that jumps (NTP correction, DST change, laptop sleep/resume, manual clock change), so history/log timestamps go backwards or duplicate, breaking ordering and 'last 30 days' retention math.
  - **Handling:** Store timestamps as UTC ISO-8601 (with offset) for human readability AND ordering, but compute DURATIONS from a monotonic clock (time.monotonic) captured at record start/stop, never from wall-clock deltas. For retention-by-age, tolerate non-monotonic wall clock (clamp negative ages to 0, never delete based on a single bad timestamp). Don't assume history file order equals chronological order — sort defensively in `yohoho history`.
- **[LOW] `schema-evolution-old-history-unreadable`** (history) — A future yohoho version changes the history record schema; old JSONL lines lack new fields or use an old shape, and the new `yohoho history` reader crashes or hides old entries.
  - **Handling:** Include a `v` schema-version field in every record and a tolerant reader that defaults missing fields and skips/upgrades unknown ones rather than crashing. Never hard-require a field added later. Keep the schema additive. Same for the log format — keep it greppable plain text so format drift never blocks reading.
- **[LOW] `logs-leak-pii-via-paths-and-env`** (logging) — Even with transcript text excluded, logs leak PII through other channels: full filesystem paths containing the username, the focused app/window title, environment dumps, or the data-dir path embedded in a diagnostics bundle the user shares publicly.
  - **Handling:** In the `yohoho doctor`/diagnostics bundle, redact home-dir paths to ~ and the username to <user>, exclude window titles entirely, and never dump environment variables. Keep the raw log on disk as-is for the user but make the SHAREABLE bundle the scrubbed one. State clearly what the bundle contains. Avoid logging the focused-app title anywhere by default.


## 7. Threading, concurrency & UI  (22 cases · 3 critical · 6 high)

- **[CRIT] `cross-thread-tk-call-crash`** (ui) — Audio thread or engine thread calls a Tk widget method (canvas.itemconfig, .configure, .update) directly instead of marshaling onto the Tk main loop.
  - **Handling:** Single-owner Tk model: the Tk main loop runs on the process MAIN thread and is the ONLY thread that ever touches widgets. Recorder/engine threads NEVER call widget methods; they push immutable event objects onto a thread-safe queue.Queue. A root.after(16, drain_queue) pump (the ~55ms/16ms animation tick already in the spec can double as the drain) is the sole consumer that applies state to the canvas. Provide a tiny UiBus.post(event) wrapper so worker code physically cannot import the widget objects.
- **[CRIT] `progress-fakes-success-on-error`** (ui) — Transcription fails (engine exception, OOM, model not loaded) but the progress bar still eases to 90% and the 'finishing' state fills it to 100% because the UI only listens for 'idle'.
  - **Handling:** Split terminal events: add an explicit 'error'/'aborted' signal distinct from 'idle'/success. The progress bar only fills to 100% + green on a SUCCESS terminal event. On error, transition to a dedicated ERROR UI state: stop easing, freeze the bar at its current fill, turn the bar/% red (#ff5454 is reserved for REC — use a separate error red or amber to stay on-brand, or invert: dim the dots and show 'failed' in muted), show a short 'failed' / 'no text' label, hold ~1.2s, then hide. State machine should be idle->recording->transcribing->(inserting->done | error).
- **[CRIT] `panel-steals-focus`** (ui) — The always-on-top Tk panel steals keyboard focus from the app the user is dictating into, so the paste lands in the panel or focus is lost.
  - **Handling:** Make the panel non-activating: Tk overrideredirect(True) + wm_attributes('-topmost', True); on macOS set wm_attributes('-type','utility') or use the NSPanel nonactivating style via the platform adapter so it never becomes key window; on Windows the platform adapter sets WS_EX_NOACTIVATE on the HWND. Never call focus_set/grab_set on the panel. Validate by asserting the previously-focused app is still frontmost right before synthesizing paste.
- **[HIGH] `rapid-toggle-state-race`** (controller) — User mashes the hotkey faster than the state machine settles, racing recording<->transcribing transitions.
  - **Handling:** Make the controller a single-threaded authority: hotkey events are also marshaled (onto the controller's own queue or the Tk queue) and applied as a strict state machine with a lock. Define legal transitions only; ignore/queue presses received in transcribing/inserting (or treat a press during transcribing as CANCEL — see cancel case). Debounce the chord matcher (~250ms) so a single physical press can't register twice. The UI derives its state ONLY from the controller's emitted state, never inferring.
- **[HIGH] `cancel-midstream-ui-teardown`** (ui) — User cancels (Esc or hotkey) while recording or mid-transcription; the UI must abort cleanly without leaving the panel or faking a paste.
  - **Handling:** Add an 'aborted' terminal event separate from success/error. Cancel sets a cancel-token the result handler checks AFTER recognize() returns (it can't preempt the blocking call) and discards the text + skips the paste/clipboard-restore-with-result path. The UI on 'aborted' shows a brief 'cancelled' in muted, no 100% fill, then hides. The controller must drop the late engine result even though the worker still completes.
- **[HIGH] `dropped-terminal-event-stuck-panel`** (controller) — The terminal status event (idle/error) is dropped or never emitted, leaving the panel stuck in transcribing forever.
  - **Handling:** Two guards: (1) wrap the entire after()-pump body in try/except that logs and reschedules itself, so one bad event never kills the loop. (2) Add a transcribing watchdog: when entering transcribing, schedule root.after(max_expected_ms based on durationSignal, e.g. clip_secs*0.4 + 8000ms ceiling) that, if still in transcribing, forces the ERROR state and hides. The recognize() worker runs with a hard timeout in the controller too.
- **[HIGH] `tray-thread-vs-tk-thread`** (ui) — pystray's event loop and Tkinter's mainloop both want to own the main thread / run their own run loop, and tray menu callbacks fire on the tray thread but touch Tk.
  - **Handling:** Tk owns the main thread and the process run loop. Run pystray in a SEPARATE thread via icon.run_detached() (Windows/Linux). On macOS, where NSStatusBar needs the main thread, prefer rumps or drive the status item through the platform adapter on the main thread interleaved with Tk, OR make the tray optional on macOS and rely on a menu-bar item managed by the adapter. Tray callbacks ONLY post events onto the UI queue (request_quit, open_settings) — never touch Tk directly. Quit posts a sentinel that the Tk thread handles: cancel after() handles, withdraw panel, root.quit().
- **[HIGH] `blocking-call-on-tk-thread`** (controller) — The blocking recognize() or the ~80-120ms paste-then-restore-clipboard sleep accidentally runs on the Tk main thread, freezing all animation.
  - **Handling:** Hard rule: the Tk main thread NEVER blocks. recognize() runs on a dedicated engine thread; the inject paste-chord + sleep + clipboard-restore runs on a worker (the controller's thread), not in an after callback. The Tk thread only runs short pump/animation work. Add a dev assertion that flags any call >16ms on the main thread in debug builds.
- **[HIGH] `tk-not-on-main-thread-macos`** (ui) — Tk root is created on a non-main thread (because the daemon spawned the UI in a worker), which is unsupported and crashes on macOS.
  - **Handling:** Architect the entry point so the process MAIN thread is reserved for Tk: main() creates the root and runs mainloop; recorder, engine, controller, and tray-detached all start as worker threads BEFORE mainloop. The daemonization (detaching/forking) must complete before Tk init, and Tk must be initialized in the final resident process's main thread. Document this as an invariant in the controller/cli design.
- **[MED] `stale-out-of-order-events`** (ui) — Status/amplitude/duration events arrive out of order or late relative to the current UI state (e.g. amplitude from a finished recording lands after transcribing started).
  - **Handling:** Stamp every event with a monotonically increasing session/generation id assigned when recording starts. The UI drains the whole queue each tick but DISCARDS any event whose generation != current. Amplitude events carry the generation; only render them in the recording state. Treat status as authoritative state transitions, amplitude/duration as advisory data valid only for the matching state+generation.
- **[MED] `panel-show-hide-race`** (ui) — Show and hide are requested in quick succession across threads, leaving the panel visible-when-should-be-hidden or vice versa.
  - **Handling:** Visibility is derived state, not imperative commands. The pump computes desired-visible = (state in {recording, transcribing, finishing, error}) each tick and calls deiconify()/withdraw() only on change. Cancel any pending hide-after handle (store the id, root.after_cancel) when a new session starts. Tag delayed hides with the generation id so a stale hide is a no-op against the current generation.
- **[MED] `topmost-fight-fullscreen`** (ui) — Panel fights other always-on-top windows or is hidden behind a fullscreen app / macOS Space, so the user gets no feedback.
  - **Handling:** Don't rely solely on visual panel for feedback: add an audible/haptic-free start cue is optional, but key is set the panel's window level to a high level (macOS NSStatusWindowLevel / collectionBehavior canJoinAllSpaces+fullScreenAuxiliary via adapter) so it shows over fullscreen Spaces. Accept that some fullscreen contexts can't be overlaid and document it; the tray icon state + a short OS notification on start/stop is the fallback feedback channel.
- **[MED] `multimonitor-positioning`** (ui) — Bottom-center positioning uses the wrong screen or wrong geometry on multi-monitor / mixed-DPI setups.
  - **Handling:** Position relative to the display containing the FOCUSED window / mouse cursor at recording start (query via the platform adapter — Windows MonitorFromWindow, macOS NSScreen for the focused app), not the Tk primary screen. Clamp the computed x,y so the full 296x80 rect stays within that monitor's visible work area. Recompute on every show, not once at startup.
- **[MED] `monitor-unplug-while-shown`** (ui) — Monitor is unplugged, resolution changes, or display arrangement changes while the panel is visible.
  - **Handling:** Bind to Tk's <Configure> on the root plus poll winfo_screenwidth/height each show; on macOS the adapter can subscribe to NSApplicationDidChangeScreenParameters and Windows WM_DISPLAYCHANGE to post a 'reposition' event onto the UI queue. On reposition, re-clamp into the current primary/focused monitor's bounds. As a cheap safety net, re-validate and re-center geometry at the start of every recording session.
- **[MED] `dpi-scaling-blurry-or-misclamped`** (ui) — HiDPI / fractional DPI scaling makes the canvas blurry or makes pixel math (dot radii, 296x80, clamping) wrong.
  - **Handling:** Query Tk scaling (root.tk.call('tk','scaling')) and winfo_fpixels('1i')/winfo_pixels to get the real scale, and size the canvas + dot radii in scaled units so dots stay crisp at 2x. On Windows the platform adapter must set per-monitor-DPI-awareness (SetProcessDpiAwarenessContext) at process start BEFORE Tk initializes. Redraw (don't bitmap-scale) on DPI change. Test at 100/150/200%.
- **[MED] `doto-font-missing`** (ui) — The bundled Doto font fails to load / register, so the wordmark, status, timer, and % render in a fallback font.
  - **Handling:** Detect at startup: list tkfont.families() after attempting registration; if 'Doto' absent, fall back to a bundled monospace (or draw the wordmark as dots on the canvas, which is on-brand and font-independent). Never hard-code the family without a guard. Register the font via the platform adapter (macOS CTFontManagerRegisterFontsForURL, Windows AddFontResourceEx, Linux copy to ~/.local/share/fonts) and verify the returned family name rather than assuming 'Doto Black'. Log which font actually resolved.
- **[MED] `after-handles-leak-on-quit`** (ui) — Pending root.after() callbacks (blink timer, waveform tick, easing tick, hide-hold) fire during or after shutdown, touching destroyed widgets.
  - **Handling:** Track all live after-ids in a set; on shutdown call after_cancel on each before root.destroy(). Guard every after-callback with an 'if not self._alive: return' check and wrap in try/except TclError. Use a single coalesced animation tick (one repeating after) rather than many independent timers so there's one thing to stop. Drive shutdown through the UI queue so it's ordered relative to other UI work.
- **[MED] `amplitude-flood-ui-starvation`** (recorder) — The sounddevice callback posts amplitude events far faster than the UI tick consumes them, flooding the queue and starving/lagging the UI.
  - **Handling:** Don't queue every amplitude. Use a single shared 'latest amplitude' slot (a float guarded by a lock, or queue maxsize=1 with drop-oldest) that the callback overwrites; the UI tick reads the current value once per frame and applies the peak-hold/decay there. This decouples audio callback rate from frame rate and bounds memory. Reserve the queue for state-transition events only.
- **[MED] `model-load-blocks-first-paint`** (engine) — First-ever recognize() triggers a lazy model load (~660MB int8 ONNX) that takes many seconds, far exceeding the progress estimate, while the panel eases to 90% and sits.
  - **Handling:** Preload/keep the model RESIDENT at daemon start (warm it with a tiny dummy buffer) so recognize() never pays load cost on the hot path. If a cold load is unavoidable, emit a distinct 'loading model' status so the UI shows 'loading…' instead of the transcribing estimate, and the watchdog uses a much larger ceiling (or is disabled) in that state. Don't let the model-load path masquerade as normal transcription.
- **[LOW] `dark-light-fixed-palette`** (ui) — The panel uses a fixed near-black palette regardless of OS dark/light mode, and translucent corners / topmost may render oddly in light mode or with reduce-transparency.
  - **Handling:** Keep the panel a self-contained opaque dark surface (don't depend on desktop color); the brand is dark by design, so don't auto-switch to light. Avoid real window transparency where possible (fake the ~2-3px rounded corners by drawing on an opaque canvas) so there's no compositor fringing. If 'reduce transparency' is set, drop any alpha and use solid colors. This is cosmetic, hence low.
- **[LOW] `duration-signal-missing-progress-broken`** (recorder) — durationSignal (clip length for the progress estimate) is missing, zero, or arrives after transcribing already started.
  - **Handling:** Make the easing purely asymptotic toward 90% (the spec's p += (0.90-p)*0.10 already needs no duration), so duration is advisory only — use it merely to pick a sane watchdog ceiling with a floor (e.g. max(8000ms, duration*0.4)). Guard against duration<=0: treat as 'unknown' and use the default ceiling. Empty/too-short recordings should short-circuit to idle with a 'too short' message before transcribing.
- **[LOW] `queue-event-coalescing-tearing`** (ui) — Multiple state events drained in one tick are applied sequentially, causing a visible flash of an intermediate state.
  - **Handling:** When draining, fold to the LAST status event of the batch for the terminal visible state (apply only the final transition), while still accumulating amplitude into the peak-hold. Compute desired visibility from the final folded state once per tick rather than toggling per event. This makes the UI converge to the latest truth without rendering intermediate flashes.


## Cross-seam gaps (caught by the critic, not by any single surface)

- FOCUS-TOKEN vs SLEEP/REVOKE seam: agents capture the focus token at record-STOP and verify before paste, and separately handle sleep/wake and TCC revocation, but nobody ties them together. The focus token must ALSO carry the session generation-id AND a 'no-sleep-boundary-crossed and permissions-still-granted' assertion. A transcription spanning a sleep/wake or a TCC revocation must invalidate the token even if the SAME app is still frontmost (slept on a terminal, woke on it hours later) - agents only guard against the app CHANGING, not against a long opaque gap on the same target.
- The 80-120ms paste->restore sleep is itself a cancel/shutdown/sleep blind spot: during that fixed sleep the modifier-release, cancel check, shutdown check, and focus re-check are all stale. No agent models a second event (cancel/SIGTERM/sleep) arriving DURING that exact post-paste/pre-restore window; the clipboard-dirty and stuck-modifier agents each own half and neither owns the interleaving.
- WARMUP recognize() on a silent buffer (cold-start agent) collides with the silence-floor guard (recorder agent): warmup deliberately feeds near-zero audio that the empty-result/silence guards are built to suppress. Nobody specifies warmup must bypass the controller/insert/history pipeline entirely and never touch clipboard, panel, or history - naive wiring makes the daemon try to paste/flash on its own startup warmup.
- GENERATION-ID lifetime across restarts vs the re-paste-last pointer and persisted history ids: multiple agents lean on a monotonic generation/session id to drop stale results, but nobody defines id lifetime across daemon restarts. A persisted history entry id, an in-flight generation id, and a re-paste-last pointer must not be confused after a restart (e.g. re-paste-last reading a generation id a fresh daemon reset to 0).
- VAD mode + CANCEL/Esc lifecycle interaction: cancel agent registers Esc only 'while a session is active' and tears it down on idle; VAD agent has VAD auto-stop a session. In VAD mode the session goes recording->transcribing WITHOUT a user stop, so the Esc-registration window is defined by VAD timing, not user action - nobody specifies who owns Esc registration/teardown when the state machine, not the user, drives the stop.
- The TRANSCRIBING WATCHDOG must use a MONOTONIC clock: watchdog ceilings (max(8s, duration*X)) measure elapsed time; a separate agent flags wall-clock skew only for history timestamps. If the watchdog measures via wall-clock and NTP steps backward mid-transcribe, it either never fires (panel stuck forever) or fires instantly (false ERROR) - the exact failure the watchdog exists to prevent.
- First-run model download is split across two agents (engine: never download on hotkey path; setup: resumable verified download) but NOBODY owns setup's download being IN PROGRESS when the user presses the hotkey, or `start` running concurrently with `setup`. setup-download and daemon-start are two processes racing the same HF cache and the same model_ready marker.
- Multi-monitor panel positioning (UI agent) and focus-token capture (injector agent) share a dependency - both need 'the focused window/its display at record start' from the platform adapter - but are speced as two independent calls. They should be ONE focus snapshot at record start (app id + window handle + display id) reused for BOTH panel placement and paste verification, so they can never disagree about what 'the target' is.
- uv/npm shim PATH resolution under a detached daemon at LOGIN autostart: the PATH-not-set agent and the autostart agents are separate. At OS-login autostart the daemon launches with a minimal login environment where ~/.local/bin and the npm global bin may not be on PATH - the exact resolution problem, but in the one context (autostart) with no interactive shell to print remediation to. Nobody connects 'shim can't find the uv-installed python' with 'this happens silently at every login'.
- Audio sample-rate/resample correctness at the recorder->engine seam: everyone assumes clean 16kHz int16 -> float32. Nobody covers a device that only supports 44.1/48kHz (very common on macOS built-in and Bluetooth), forcing a resample to 16k; a wrong or skipped resample feeds the model audio at the wrong rate, producing confident garbage that looks like a model-accuracy problem - and it passes the RMS/silence guards.
